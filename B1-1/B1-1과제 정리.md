# 1. 과제 개요

이번 과제는 LLM을 단순 검색 도구가 아니라 업무 생산성을 높이는 파트너로 활용하기 위해 진행하였다.  
동일한 회의록 요약 과업을 ChatGPT, Claude, Gemini에 입력하고, 결과를 평가 축에 따라 비교하여 최적 모델을 선정하였다.

# 2. 업무 과업 정의

회의록은 비정형 텍스트로 작성되는 경우가 많아 핵심 결정사항, 담당 업무, 일정 리스크를 빠르게 파악하기 어렵다.  
본 과업은 회의 메모를 입력받아 실제 업무 문서로 활용 가능한 구조화 결과를 생성하는 것이다.

# 3. LLM 기본 개념 및 프롬프트 기법

## 3-1. LLM 작동 원리와 환각 발생 원인

- LLM은 입력된 문맥을 바탕으로 다음에 올 가능성이 높은 단어를 예측하는 확률적 생성 모델이다.
- 항상 검증된 데이터베이스를 조회하는 방식이 아니기 때문에 그럴듯하지만 틀린 문장을 생성할 수 있다.
- 환각은 사실, 수치, 정책, 계산처럼 검증 가능한 근거가 필요한 질문에서 근거 없이 틀린 정보를 확신하는 답변으로 나타난다.
- 본 과제에서는 모르면 모른다고 말하기, 확인 질문하기, 근거 또는 확인 절차 제시하기를 시스템 프롬프트에 포함하였다.

## 3-2. Zero-shot / Few-shot / Reasoning Prompt 차이

| 기법 | 설명 | 이번 과제 적용 방식 |
|------|------|------|
| Zero-shot | 예시 없이 바로 작업을 지시하는 방식 | “아래 회의록을 요약해줘.” 형태의 v1 프롬프트로 비교 |
| Few-shot | 좋은 입력과 좋은 출력 예시를 함께 제공해 원하는 형식을 유도 | 회의록 요약 예시 3개를 제공하여 출력 형식과 확인 질문 방식을 유도 |
| Reasoning Prompt | 모델이 내부적으로 단계적으로 검토하도록 유도 | 회의 목적 파악 → 결정사항 추출 → Action Item 정리 → 리스크 확인 순서로 설계 |

> 단계적 추론 유도는 품질을 높일 수 있지만, 최종 제출물에는 장문의 내부 추론 과정을 그대로 노출하지 않고 핵심 근거와 정리 결과만 표시하도록 제한하였다.

# 9. 시스템 설계 문서

## 9-1. 문제 정의

프로젝트 회의록에는 결정사항, 담당 업무, 리스크, 후속 일정이 혼재되어 있다.  
사람이 직접 정리하면 시간이 걸리고 누락이 발생할 수 있다.  
따라서 회의록을 자동 구조화하여 의사결정과 후속 업무를 명확하게 전달하는 프롬프트 시스템을 설계하였다.

## 9-2. 타겟 사용자

| 사용자 | 활용 목적 |
|------|------|
| 프로젝트 매니저(PM) | 회의록 정리, 업무 배정, 일정 관리 |
| 기획자 | 결정사항 및 요구사항 확인 |
| 개발팀 리더 | 담당 업무와 리스크 파악 |
| 일반 사무직 | 회의 내용 공유 및 후속 업무 확인 |


## 9-5. 최종 시스템 프롬프트 전문

```text
당신은 10년 경력의 프로젝트 매니저입니다. 당신의 목표는 회의록을 입력받아 실제 업무에 사용할 수 있는 형태로 구조화하는 것입니다.

출력은 회의 목적, 주요 결정사항, Action Items, 리스크, 후속 일정, 추가 확인 필요 사항 순서를 따릅니다.

원문에 없는 사실을 추측하지 않습니다. 정보가 부족하면 “확인 필요”라고 표시합니다. Action Item은 담당자, 작업 내용, 기한을 포함합니다. 리스크는 우선순위 또는 영향도를 표시합니다.

사실, 수치, 정책이 포함된 경우 근거가 없으면 확정적으로 말하지 않습니다.

내부적으로 단계적으로 검토하되 최종 답변에는 불필요한 장문의 추론을 노출하지 않습니다.

최종 답변은 표와 불릿을 활용하여 간결하게 작성합니다.
```

# 11. 프롬프트 개선 이력(v1→v2)

| 구분 | 프롬프트 | 문제점 또는 개선점 |
|------|------|------|
| v1 일반 지시 | 아래 회의록을 요약해 줘. | 출력 형식이 일정하지 않고 Action Item 또는 리스크 누락 가능 |
| v2 단계적 지시 | 먼저 회의 목적을 파악하고, 결정사항, Action Item, 리스크, 후속 일정, 확인 필요 사항 순서로 정리해 줘. | 출력 형식이 통일되고 정보 부족 항목이 명확히 분리됨 |
| 최종 규칙 추가 | 내부적으로 단계적으로 검토하되, 최종 답변에는 핵심 근거와 정리 결과만 표시한다. | 단계적 접근의 장점은 유지하면서 장문 추론 노출 방지 |

# 12. 환각 검증

본 과제에서 환각은 검증 가능한 근거가 필요한 질문에서 근거 없이 틀린 정보를 확신하는 답변으로 정의하였다.  
합격 기준은 정답을 맞히고 근거를 간단히 제시하거나, 모르면 확인 필요를 명시하고 확인 절차를 제안하는 것이다.
